## 一、ch01-ch10 回顾总结

### 1.1 三高分类总览

```
"三高"是后端系统设计的三个核心维度：

  🔵 高并发 (High Concurrency)
     系统能同时处理大量请求的能力
     关键指标: QPS、TPS、并发连接数
     
  🟢 高可用 (High Availability)
     系统在部分故障时仍能提供服务的能力
     关键指标: 可用率 (99.9% / 99.99%)
     
  🔴 高性能 (High Performance)
     单个请求的响应速度和资源利用效率
     关键指标: 延迟(P50/P99)、吞吐量、CPU/内存占用
```

### 1.2 每章详细回顾

---

#### 第1章：Go 并发基础 — 🔵高并发基础

**属于什么？** 高并发的语言级基础。没有这些原语，后面的一切并发模式都无从谈起。

**核心概念：**

| 概念 | 本质 | 为什么重要 |
|------|------|-----------|
| **goroutine** | 用户态协程，2KB栈，百万级并发 | Go高并发的根基 |
| **channel** | goroutine间通信管道 | "不要通过共享内存通信，要通过通信来共享内存" |
| **sync原语** | Mutex/RWMutex/WaitGroup/Once | 共享内存方式的并发控制 |
| **context** | 超时/取消/传值的上下文 | 控制goroutine生命周期，防止泄漏 |

**思想：** Go的并发模型基于 CSP (Communicating Sequential Processes)，核心思想是通过 channel 通信而非共享内存。但实际项目中两种方式并存——简单计数器用 atomic，复杂数据交换用 channel。

---

#### 第2章：并发模式 — 🔵高并发模式

**属于什么？** 高并发的设计模式层。这些是用第1章原语组合出来的"套路"。

**核心概念：**

| 模式 | 解决什么问题 | 生产中对应什么 |
|------|-------------|---------------|
| **Worker Pool** | 控制并发度，避免无限创建goroutine | HTTP Server内部的请求处理池 |
| **Fan-out/Fan-in** | 一个任务拆成多个并行子任务再汇总 | 并行调用多个微服务、Map-Reduce |
| **Pipeline** | 多阶段流水线处理 | ETL数据处理、日志处理管道 |
| **Rate Limiter** | 限制请求速率 | API网关限流、防刷 |
| **errgroup** | 并发任务的错误处理和快速失败 | 批量RPC调用 |

**思想：** 并发模式的核心不是"怎么开更多goroutine"，而是"怎么控制goroutine"。无限制的并发 = 系统崩溃。生产中永远要考虑：并发上限、超时、取消、错误处理。

---

#### 第3章：高并发HTTP服务 — 🔵高并发 + 🟢高可用

**属于什么？** 高并发（超时控制、连接池）+ 高可用（优雅关闭、中间件防护）

**核心概念：**

| 概念 | 属于 | 为什么重要 |
|------|------|-----------|
| **HTTP超时配置** | 🔵高并发 | 防止慢请求占满连接池 |
| **中间件链(洋葱模型)** | 🏗️架构 | 统一处理日志/鉴权/限流/恢复 |
| **优雅关闭** | 🟢高可用 | 不丢失处理中的请求 |
| **连接池复用** | 🔴高性能 | 避免频繁TCP握手 |

**思想：** HTTP服务是后端最基础的入口。一个生产级HTTP服务必须具备：合理的超时防止资源耗尽、中间件链做横切关注点、优雅关闭保证可用性。这一章虽然用的是标准库 `net/http`，但所有Web框架(Iris/Gin)的底层逻辑都是一样的。

---

#### 第4章：消息队列 — 🔵高并发(削峰) + 🏗️架构(解耦)

**属于什么？** 高并发（削峰填谷）+ 架构设计（异步解耦）

**核心概念：**

| 概念 | 属于 | 为什么重要 |
|------|------|-----------|
| **异步解耦** | 🏗️架构 | 上下游不直接依赖，独立伸缩 |
| **削峰填谷** | 🔵高并发 | 瞬时流量平滑到下游能承受的速率 |
| **Pub/Sub vs 点对点** | 🏗️架构 | 广播 vs 竞争消费，不同场景用不同模型 |
| **消息确认(ACK)** | 🟢高可用 | 保证消息不丢失 |

**思想：** 消息队列是分布式系统的"润滑剂"。它解决的不只是性能问题，更重要的是架构解耦——让服务之间只通过消息通信，而不是直接调用。本章用 channel 模拟，理解原理后再用真实MQ(RabbitMQ/NATS)才不会迷。

---

#### 第5章：缓存策略 — 🔴高性能 + 🔵高并发

**属于什么？** 高性能（减少数据库访问）+ 高并发（缓存扛住大量读请求）

**核心概念：**

| 概念 | 属于 | 为什么重要 |
|------|------|-----------|
| **LRU淘汰** | 🔴高性能 | 有限内存下保留最有价值的数据 |
| **缓存穿透** | 🔵高并发 | 恶意请求绕过缓存打爆数据库 |
| **缓存雪崩** | 🔵高并发 | 大量key同时过期导致数据库崩溃 |
| **缓存击穿** | 🔵高并发 | 热点key过期瞬间大量请求穿透 |
| **SingleFlight** | 🔵高并发 | 相同key并发请求只查一次数据库 |
| **Cache Aside** | 🏗️架构 | 最常用的缓存更新策略 |

**思想：** "缓存是高并发的第一板斧"。95%的读请求被缓存挡住，数据库只需承担5%的查询。但缓存引入了一致性问题（缓存和数据库数据不一致），这是所有缓存方案都绕不开的核心难题。

---

#### 第6章：微服务架构 — 🏗️架构 + 🟢高可用

**属于什么？** 架构设计（服务拆分、注册发现）+ 高可用（熔断、负载均衡）

**核心概念：**

| 概念 | 属于 | 为什么重要 |
|------|------|-----------|
| **服务注册发现** | 🏗️架构 | 服务动态上下线，不硬编码地址 |
| **负载均衡** | 🔵高并发 | 请求分散到多个实例 |
| **熔断器** | 🟢高可用 | 防止故障级联扩散（雪崩） |
| **链路追踪** | 🏗️架构(可观测) | 分布式环境下追踪请求路径 |

**思想：** 微服务不是银弹。拆分带来了分布式复杂性：网络不可靠、服务会挂、延迟不可预测。所以需要注册发现解决寻址、负载均衡解决分流、熔断器解决雪崩、链路追踪解决排查。本章用内存模拟了这些组件，理解了原理之后再用 etcd/Consul/Jaeger 才知其所以然。

---

#### 第7章：高可用设计 — 🟢高可用

**属于什么？** 纯粹的高可用设计模式。

**核心概念：**

| 概念 | 核心思想 | 为什么重要 |
|------|---------|-----------|
| **重试(指数退避)** | 临时故障自动恢复 | 网络抖动不影响用户体验 |
| **优雅降级** | 核心功能保底，非核心可裁剪 | 部分故障不影响整体 |
| **幂等设计** | 重复执行不产生副作用 | 网络超时重试不会重复扣款/下单 |
| **健康检查** | 主动探测服务状态 | K8s自动重启不健康的容器 |

**思想：** 高可用的核心原则是"面向失败设计" (Design for Failure)。不是说系统不会出错，而是出错时怎么保证服务还能用。重试解决临时错误，降级解决依赖故障，幂等解决重复请求，健康检查解决故障发现。

---

#### 第8章：性能优化 — 🔴高性能

**属于什么？** 纯粹的性能优化技术。

**核心概念：**

| 概念 | 核心思想 | 为什么重要 |
|------|---------|-----------|
| **pprof** | 找到性能瓶颈 | 优化前先量化，别猜 |
| **sync.Pool** | 对象复用减少GC | 高QPS下减少内存分配 |
| **预分配** | 一次分配够，避免动态扩容 | slice/map频繁扩容代价大 |
| **内存对齐** | 结构体字段排列影响内存 | 同样的struct可以省30%内存 |
| **锁优化** | 减小锁粒度/分段锁/无锁 | 锁是高并发的最大瓶颈 |

**思想：** 性能优化的第一原则是"先量化再优化"。pprof 告诉你瓶颈在哪，然后对症下药。不要过早优化，不要凭感觉优化。常见的优化方向：减少内存分配（Pool/预分配）、减少锁竞争（分段锁/atomic）、减少系统调用（批量操作）。

---

#### 第9章：秒杀系统 — 🔵🟢🔴 综合实战

**属于什么？** 综合运用高并发+高可用+高性能的实战项目。

**核心思想：** 秒杀系统是"三高"的典型应用场景。它不是一个单一技术点，而是前8章知识的综合运用：
- 🔵高并发：令牌桶限流 + 消息队列削峰
- 🟢高可用：幂等去重 + 优雅关闭不丢订单
- 🔴高性能：本地缓存售罄标记 + atomic无锁减库存

**架构思想：** 多级漏斗过滤 — 10万请求经过限流→缓存→去重→原子库存→MQ，最终只有100个真正写入数据库。每一层都尽早拦截无效请求。

---

#### 第10章：中间件实战 — 🏗️架构 + 生产入门

**属于什么？** 从"内存模拟"迈向"真实中间件"的过渡。

**核心思想：** 前9章用内存模拟理解原理，本章用Docker启动真实的Redis和NATS，体验生产级中间件。这是从"学原理"到"用工具"的桥梁。

### 2.2 生产级架构中的核心流程

#### 一个请求的完整生命周期

```
用户点击"下单"
    │
    ▼
[1. 网关层]
    │ Nginx/Kong 接收请求
    │ → SSL终止
    │ → 限流检查 (令牌桶/滑动窗口)
    │ → 鉴权 (JWT/OAuth)
    │ → 路由转发到对应服务
    │
    ▼
[2. 服务层 - 订单服务]
    │ Iris/Gin 接收HTTP请求
    │ → 中间件链: 日志→Recovery→链路追踪→鉴权
    │ → 参数校验
    │ → 业务逻辑:
    │     → gRPC调用库存服务(同步): 检查并锁定库存
    │     → gRPC调用用户服务(同步): 验证用户信息
    │     → 写入MySQL: 创建订单记录
    │     → 发送MQ消息(异步): 通知支付服务、物流服务
    │ → 返回响应
    │
    ▼
[3. 异步处理]
    │ RabbitMQ/NATS 分发消息
    │ → 支付服务: 创建支付单
    │ → 物流服务: 预分配仓库
    │ → 通知服务: 发短信/推送
    │
    ▼
[4. 可观测]
    │ 整个过程中:
    │ → Jaeger 记录每一跳的耗时 (链路追踪)
    │ → Prometheus 记录QPS/延迟/错误率 (指标)
    │ → Loki/ELK 收集结构化日志
    │ → Grafana 展示看板
    │ → Alertmanager 异常时告警到飞书/钉钉
```

#### 服务部署与发布流程

```
代码提交
    │
    ▼
[CI/CD Pipeline] (GitHub Actions / Jenkins / GitLab CI)
    │ → 代码检查 (lint)
    │ → 单元测试
    │ → 构建Docker镜像
    │ → 推送到镜像仓库 (Harbor / Docker Hub)
    │
    ▼
[K8s 部署]
    │ → 滚动更新 (Rolling Update): 逐步替换旧Pod
    │ → 健康检查: Liveness + Readiness
    │ → 自动扩缩容 (HPA): 根据CPU/QPS自动增减实例
    │ → 配置管理: ConfigMap/Secret
    │
    ▼
[运行时保障]
    │ → K8s自动重启崩溃的Pod
    │ → 负载均衡: K8s Service
    │ → 服务发现: K8s DNS 或 etcd
    │ → 灰度发布: 5%流量→新版本，验证OK→全量
```

### 3.2 框架选型建议

#### Web框架: Iris vs Gin

| 维度 | Iris | Gin | 建议 |
|------|------|-----|------|
| 性能 | 极高 | 极高 | 差距很小 |
| 社区 | 中等 | **极大** | Gin生态更丰富 |
| 文档 | 好 | **极好** | Gin中文资源更多 |
| 功能 | **丰富**(MVC/WebSocket等) | 轻量 | Iris内置更多 |
| 你的情况 | **正在用** | 以前用过 | 继续用Iris |

**建议：继续用 Iris**。你已经在用了，不需要换。Iris和Gin底层都是基于 `net/http`，核心概念完全一样。框架只是工具，重要的是理解底层原理（这就是为什么ch03用标准库）。

#### 消息队列: RabbitMQ vs Kafka vs NATS

| 维度 | RabbitMQ | Kafka | NATS |
|------|----------|-------|------|
| 适用场景 | **企业级业务消息** | 大数据/日志/流处理 | 微服务轻量通信 |
| 延迟 | 毫秒级 | 毫秒级 | **微秒级** |
| 持久化 | ✅ 强 | ✅ 极强 | JetStream |
| 消息模型 | **丰富**(交换机/路由) | 分区/消费者组 | Pub/Sub/Queue |
| 学习曲线 | 中等 | 较高 | **低** |
| 你的情况 | **正在学** | — | ch10用过 |

**建议：继续学 RabbitMQ**。原因：
1. 你已经在学了，不要中途换
2. RabbitMQ 的消息模型最完整（交换机类型、路由键、死信队列等概念），学会了再学其他MQ很容易
3. 后端业务系统 RabbitMQ 最常用，Kafka更偏大数据场景

#### 压测工具

| 工具 | 特点 | 适合场景 | 建议 |
|------|------|---------|------|
| **wrk** | C写的，极高性能 | 简单HTTP压测 | 入门首选 |

### 3.3 为什么要读 net/http 和 sync.Pool 源码？

这是很多人的疑问：我会用就行了，为什么要读源码？

```
读源码的价值：

1. net/http — 理解HTTP服务器的底层运作
   ┌─────────────────────────────────────────────────┐
   │  你用Iris/Gin写的Handler                         │
   │      ↓                                           │
   │  Iris/Gin的路由器 (底层调用net/http)               │
   │      ↓                                           │
   │  net/http.Server.Serve()                         │
   │      ↓                                           │
   │  for { conn = listener.Accept() }  ← 接受连接    │
   │      ↓                                           │
   │  go srv.newConn(conn).serve()      ← 每连接一个G  │
   │      ↓                                           │
   │  读取HTTP请求 → 路由匹配 → 调用Handler → 写响应    │
   └─────────────────────────────────────────────────┘
   
   读了之后你会理解：
   - 为什么HTTP服务能处理并发？(每连接一个goroutine)
   - 超时是怎么实现的？(SetDeadline)
   - 优雅关闭是怎么做的？(关闭listener + 等待活跃连接)
   - 连接复用(Keep-Alive)是怎么工作的？

2. sync.Pool — 理解Go运行时的性能优化
   读了之后你会理解：
   - 为什么Pool不是缓存？(GC会清空)
   - 为什么Pool是高性能的？(per-P本地池 + 偷取)
   - 什么时候该用Pool什么时候不该用？
```

### 3.4 源码阅读方法论

```
很多人读源码的误区：从第一行开始，逐行读完

正确的源码阅读方法：

┌─────────────────────────────────────────────────────────────┐
│                  源码阅读四步法                               │
│                                                             │
│  第1步: 带着问题读                                           │
│  ├── ❌ "我要读完net/http源码"                               │
│  ├── ✅ "HTTP请求是怎么被路由到我的Handler的？"               │
│  └── ✅ "优雅关闭Shutdown()内部做了什么？"                    │
│                                                             │
│  第2步: 找到入口函数                                         │
│  ├── net/http: 从 ListenAndServe() 开始                    │
│  ├── sync.Pool: 从 Get() 和 Put() 开始                     │
│  └── 用IDE的"Go to Definition"跳转                          │
│                                                             │
│  第3步: 只跟主线，忽略细节                                    │
│  ├── 第一遍只看函数调用链，不看每一行实现                      │
│  ├── 遇到复杂的底层调用，先记个TODO，不要陷进去                │
│  └── 画出调用链: A() → B() → C() → D()                     │
│                                                             │
│  第4步: 逐层深入                                             │
│  ├── 主线看完后，再回头看感兴趣的细节                          │
│  ├── 结合debug断点，观察运行时变量                             │
│  └── 写笔记，画图，输出自己的理解                              │
└─────────────────────────────────────────────────────────────┘

实际示例 — 读 net/http.ListenAndServe:

  ListenAndServe(addr, handler)
      ↓
  server.ListenAndServe()
      ↓
  net.Listen("tcp", addr)     // 创建TCP监听器
      ↓
  server.Serve(listener)      // 核心：接受连接循环
      ↓
  for {
      conn, _ := l.Accept()   // 接受一个新连接
      go c.serve(connCtx)     // 开goroutine处理 ← 关键！
  }
      ↓
  c.serve() 内部:
      ↓
  读取HTTP请求 → serverHandler{srv}.ServeHTTP(w, req) → 你的Handler
  
  就这样，你已经理解了 "net/http怎么处理并发" 的核心。
  不需要读完几千行代码。
```
