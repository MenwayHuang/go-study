# 第5章：缓存策略

## 🎯 学习目标

掌握高并发系统中缓存的使用方法，理解缓存穿透、雪崩、击穿的原理和防护方案。

## 📚 为什么缓存是高并发的第一板斧？

```
没有缓存:
  用户请求 → 应用服务器 → 数据库 (10ms/请求, 最大1000QPS)
  10000 QPS → 数据库扛不住 → 服务崩溃

有缓存:
  用户请求 → 应用服务器 → 缓存命中? ──是──→ 返回 (0.5ms)
                              │
                              否
                              ↓
                           数据库 → 写入缓存 → 返回

  缓存命中率95%: 10000 QPS × 5% = 500次 数据库查询, 轻松扛住
```

## 5.2 缓存三大问题详解

### 缓存穿透

```
  攻击者请求不存在的 ID: /user/99999999
  
  请求 → 缓存 Miss → 数据库 Miss → 返回空
  请求 → 缓存 Miss → 数据库 Miss → 返回空  (每次都打到数据库!)
  
  防护方案:
  1. 空值缓存: 查不到也缓存一个空值 (TTL短一些，如30秒)
  2. 布隆过滤器: 预先加载所有合法ID，不在过滤器里的直接拒绝
  3. 参数校验: ID格式不对的直接拒绝
```

### 缓存雪崩

```
  大量 key 在同一时刻过期
  
  T=0: 缓存正常，数据库压力小
  T=1h (大量key过期): 请求全打到数据库 → 数据库崩溃
  
  防护方案:
  1. 随机过期: TTL = baseTTL + random(0, 300秒)
  2. 永不过期 + 异步刷新: 缓存不设过期，后台定时刷新
  3. 多级缓存: 本地缓存 + Redis，即使Redis雪崩也有本地兜底
```

### 缓存击穿

```
  热点 key 突然过期，大量并发同时查询
  
  key="热门商品" 过期
  1000个并发请求同时 → 缓存 Miss → 1000个请求同时打到数据库
  
  防护方案:
  1. singleflight: 相同key的并发请求只有一个去查数据库，其余等待结果
  2. 互斥锁: 只有拿到锁的请求去查，其余等待
  3. 热点key永不过期 + 异步刷新
```

## 5.4 Cache Aside 模式 (最常用)

```
  读操作:
  1. 先读缓存
  2. 缓存命中 → 返回
  3. 缓存未命中 → 读数据库 → 写入缓存 → 返回

  写操作:
  1. 先更新数据库
  2. 再删除缓存 (注意: 是删除，不是更新!)
  
  为什么删除而不是更新缓存?
  - 避免并发写导致缓存和数据库不一致
  - 删除后下次读取会自动回填最新数据
```
