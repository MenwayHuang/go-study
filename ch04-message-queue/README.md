# 第4章：消息队列解耦

## 🎯 学习目标

理解消息队列在高并发系统中的核心作用，掌握异步处理、削峰填谷、服务解耦的实现方式。

## 📚 为什么需要消息队列？

### 同步 vs 异步

```
❌ 同步调用 (所有步骤串行，用户等待时间长):

  用户下单 → 创建订单(50ms) → 扣库存(50ms) → 发通知(100ms) → 记日志(30ms)
  总耗时: 230ms，用户一直在等

✅ 异步解耦 (核心操作同步，非核心异步):

  用户下单 → 创建订单(50ms) → 扣库存(50ms) → 返回成功 (100ms)
                                    ↓
                              消息队列 (异步)
                              ├── 发通知服务
                              ├── 记日志服务
                              └── 积分服务
  用户等待: 100ms，体验提升50%+
```

### 消息队列三大核心价值

```
1. 解耦 (Decoupling)
   ┌─────────┐     ┌──────┐     ┌─────────┐
   │ 订单服务 │────→│  MQ  │────→│ 通知服务 │
   └─────────┘     └──────┘     └─────────┘
   订单服务不需要知道谁消费消息，新增消费者不影响生产者

2. 削峰填谷 (Peak Shaving)
   请求量:  ████████████                          ← 高峰 10000 QPS
   处理量:  ████ ████ ████ ████ ████ ████         ← 匀速 2000 QPS
   队列:   [消息堆积，慢慢消费]

3. 最终一致性 (Eventual Consistency)
   即使下游暂时不可用，消息不丢失，恢复后继续处理
```

## 📚 知识点总览

```
消息队列
├── 4.1 内存消息队列 (不依赖外部组件，纯Go实现)
│   ├── 基于 channel 的简单MQ
│   ├── 支持多Topic
│   ├── 支持多消费者组
│   └── 消息确认(ACK)机制
├── 4.2 发布/订阅模式 (Pub/Sub)
│   ├── 一条消息多个消费者
│   └── 广播 vs 组内竞争
├── 4.3 可靠消息投递
│   ├── 消息持久化
│   ├── 消息重试
│   └── 死信队列 (DLQ)
└── 4.4 实战: 订单异步处理系统
    ├── 生产者: HTTP接收订单
    ├── 消费者: Worker消费处理
    └── 监控: 队列长度/消费速率
```

## 4.1 为什么先学内存MQ？

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **内存MQ (本章)** | 零依赖、学习原理 | 不持久化、单进程 | 学习、单体应用内 |
| Redis Stream | 轻量、持久化 | 功能不如专业MQ | 中小规模、已有Redis |
| NATS | 超高性能、云原生 | 生态不如Kafka | 微服务通信 |
| Kafka | 高吞吐、持久化 | 运维复杂、重 | 大数据、日志收集 |
| RabbitMQ | 功能丰富、协议标准 | 性能一般 | 企业级、复杂路由 |

**学习路径**: 内存MQ理解原理 → Redis Stream实际用 → NATS/Kafka生产级
