# 第4章：消息队列解耦

## 🎯 学习目标

理解消息队列在高并发系统中的核心作用，掌握异步处理、削峰填谷、服务解耦的实现方式。

## 📚 为什么需要消息队列？

### 同步 vs 异步

```
❌ 同步调用 (所有步骤串行，用户等待时间长):

  用户下单 → 创建订单(50ms) → 扣库存(50ms) → 发通知(100ms) → 记日志(30ms)
  总耗时: 230ms，用户一直在等

✅ 异步解耦 (核心操作同步，非核心异步):

  用户下单 → 创建订单(50ms) → 扣库存(50ms) → 返回成功 (100ms)
                                    ↓
                              消息队列 (异步)
                              ├── 发通知服务
                              ├── 记日志服务
                              └── 积分服务
  用户等待: 100ms，体验提升50%+
```

### 消息队列三大核心价值

```
1. 解耦 (Decoupling)
   ┌─────────┐     ┌──────┐     ┌─────────┐
   │ 订单服务 │────→│  MQ  │────→│ 通知服务 │
   └─────────┘     └──────┘     └─────────┘
   订单服务不需要知道谁消费消息，新增消费者不影响生产者

2. 削峰填谷 (Peak Shaving)
   请求量:  ████████████                          ← 高峰 10000 QPS
   处理量:  ████ ████ ████ ████ ████ ████         ← 匀速 2000 QPS
   队列:   [消息堆积，慢慢消费]

3. 最终一致性 (Eventual Consistency)
   即使下游暂时不可用，消息不丢失，恢复后继续处理
```

## 4.1 消息队列核心概念详解

### 消息队列 vs 发布订阅

```
消息队列 (Message Queue) — 点对点模式:
  Producer ──→ [Queue] ──→ Consumer1
                      ╲──→ Consumer2 (竞争消费，一条消息只被一个消费者处理)

发布订阅 (Pub/Sub) — 广播模式:
  Publisher ──→ [Topic] ──→ Subscriber1 (收到消息)
                        ──→ Subscriber2 (也收到同一条消息)
                        ──→ Subscriber3 (也收到同一条消息)
```

**关键区别**：
| 特性 | 消息队列 | 发布订阅 |
|------|----------|----------|
| 消费方式 | 竞争消费（负载均衡） | 广播（每个订阅者都收到） |
| 消息命运 | 被一个消费者处理后消失 | 被所有订阅者各自处理 |
| 典型场景 | 任务分发、订单处理 | 事件通知、配置同步 |

### 消息队列的核心组件

```
┌─────────────────────────────────────────────────────────────┐
│                      Message Queue                          │
│                                                             │
│  ┌──────────┐    ┌─────────────────────┐    ┌──────────┐   │
│  │ Producer │───→│   Topic/Queue       │───→│ Consumer │   │
│  │ (生产者)  │    │   [msg1][msg2][msg3]│    │ (消费者)  │   │
│  └──────────┘    └─────────────────────┘    └──────────┘   │
│                           ↑                                 │
│                      Buffer/持久化                          │
└─────────────────────────────────────────────────────────────┘
```

- **Producer (生产者)**：发送消息的一方，只需知道 Topic 名称
- **Consumer (消费者)**：接收消息的一方，从 Topic 拉取或被推送消息
- **Topic/Queue**：消息的逻辑分类，如 "orders"、"notifications"
- **Buffer**：缓冲区，用于削峰填谷，生产快于消费时暂存消息

### 消息确认机制 (ACK)

```
无ACK (At-most-once):
  Consumer 取走消息 → 消息立即删除 → Consumer 崩溃 → 消息丢失！

有ACK (At-least-once):
  Consumer 取走消息 → 消息标记"处理中" → Consumer 处理完发ACK → 消息删除
                                      → Consumer 崩溃 → 消息重新投递
```

**三种投递语义**：
| 语义 | 含义 | 实现方式 | 风险 |
|------|------|----------|------|
| At-most-once | 最多一次 | 发完就删 | 可能丢消息 |
| At-least-once | 至少一次 | ACK后删 | 可能重复消费 |
| Exactly-once | 恰好一次 | 事务+去重 | 实现复杂、性能差 |

**生产建议**：大多数场景用 At-least-once + 消费者幂等处理

### 为什么先学内存MQ？

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **内存MQ (本章)** | 零依赖、学习原理 | 不持久化、单进程 | 学习、单体应用内 |
| Redis Stream | 轻量、持久化 | 功能不如专业MQ | 中小规模、已有Redis |
| NATS | 超高性能、云原生 | 生态不如Kafka | 微服务通信 |
| Kafka | 高吞吐、持久化 | 运维复杂、重 | 大数据、日志收集 |
| RabbitMQ | 功能丰富、协议标准 | 性能一般 | 企业级、复杂路由 |

**学习路径**: 内存MQ理解原理 → Redis Stream实际用 → NATS/Kafka生产级

### 本章代码实现要点

#### mq.go — 基于 channel 的消息队列
```go
// 核心数据结构
type MessageQueue struct {
    topics map[string]chan *Message  // 每个Topic一个channel
}

// 发布：往channel发消息，满了就报错（或阻塞/丢弃）
// 订阅：返回channel，多个消费者共享同一个channel = 竞争消费
```

#### pubsub.go — 发布订阅
```go
// 核心数据结构
type PubSub struct {
    subscribers map[string][]chan *Message  // 每个订阅者有自己的channel
}

// 发布：遍历所有订阅者的channel，逐个发送 = 广播
// 订阅：创建新channel加入列表
```
