## 📚 高可用核心思想

```
高可用 = 系统在部分故障时仍能提供服务

可用性指标:
  99%    → 年停机 3.65天   (基本不可接受)
  99.9%  → 年停机 8.76小时  (普通系统)
  99.99% → 年停机 52.6分钟  (高可用系统)
  99.999%→ 年停机 5.26分钟  (金融级系统)
```

## 7.1 重试策略详解

### 重试策略对比

```
固定间隔:     |--1s--|--1s--|--1s--|
指数退避:     |--1s--|---2s---|------4s------|
指数退避+抖动: |--0.8s--|---2.3s---|------3.7s------|

为什么需要抖动?
  1000个请求同时失败 → 1秒后同时重试 → 又全部失败 (重试风暴!)
  加了抖动后，重试时间分散开，避免风暴
```

### 指数退避算法原理

```go
// 指数退避公式
delay = baseDelay * (2 ^ retryCount)

// 带抖动的指数退避
delay = baseDelay * (2 ^ retryCount) * (0.5 + random(0, 0.5))

// 示例: baseDelay=1s
// 第1次重试: 1s * 2^1 = 2s  (带抖动: 1s~2s)
// 第2次重试: 1s * 2^2 = 4s  (带抖动: 2s~4s)
// 第3次重试: 1s * 2^3 = 8s  (带抖动: 4s~8s)
```

### 重试的注意事项

| 要点 | 说明 |
|------|------|
| **最大重试次数** | 必须设置上限，避免无限重试 |
| **最大延迟** | 指数增长要封顶，如最大30秒 |
| **可重试错误** | 只重试临时错误（超时、503），不重试业务错误（400、404） |
| **幂等性** | 重试的操作必须是幂等的，否则可能重复执行 |
| **熔断配合** | 重试多次失败后应触发熔断，避免持续重试 |

## 7.2 优雅降级策略

### 什么时候需要降级？

```
正常情况:
  用户请求 → 推荐服务 → 返回个性化推荐

推荐服务挂了:
  ❌ 无降级: 用户请求 → 推荐服务超时 → 报错 → 用户体验差
  ✅ 有降级: 用户请求 → 推荐服务超时 → 返回热门商品(兜底) → 用户体验可接受
```

### 常见降级策略

| 策略 | 实现 | 适用场景 |
|------|------|----------|
| **返回默认值** | 服务不可用时返回预设值 | 非核心功能 |
| **缓存兜底** | 返回上次成功的缓存数据 | 数据时效性要求不高 |
| **功能裁剪** | 关闭非核心功能 | 系统过载时 |
| **读写降级** | 只读不写，或异步写 | 数据库压力大时 |

## 7.3 幂等设计

```
什么是幂等?
  同一个操作执行一次和执行多次，效果相同

  幂等:   GET /user/1       (读取，天然幂等)
  幂等:   PUT /user/1 {}    (更新为固定值，幂等)
  非幂等: POST /orders      (创建，每次创建新订单)
  非幂等: POST /account/pay (扣款，每次都扣钱!)

  网络超时 → 客户端不知道服务端是否收到 → 重试 → 可能重复执行
  所以需要幂等设计来保证重复请求不会产生副作用
```
