## 6.1 服务注册与发现原理详解

### 为什么需要服务发现？

```
❌ 硬编码服务地址:
   orderService.Call("192.168.1.100:8080")
   问题: 服务IP变了怎么办? 服务扩容了怎么办? 服务挂了怎么办?

✅ 服务发现:
   orderService.Call("order-service")  // 只需要服务名
   注册中心自动解析为可用的 IP:Port 列表
```

### 服务注册发现流程

```
┌─────────────────────────────────────────────────────────────┐
│                     服务注册发现流程                          │
│                                                             │
│  1. 服务启动时注册                                           │
│     Service A ──Register──→ Registry                        │
│     (name="user-svc", addr="10.0.0.1:8080")                │
│                                                             │
│  2. 定时心跳保活                                             │
│     Service A ──Heartbeat──→ Registry (每5秒)               │
│     超时未收到心跳 → 标记为不健康/剔除                         │
│                                                             │
│  3. 服务发现                                                 │
│     Client ──Discover("user-svc")──→ Registry               │
│     Registry ──返回──→ ["10.0.0.1:8080", "10.0.0.2:8080"]   │
│                                                             │
│  4. 负载均衡选择一个地址调用                                   │
│     Client ──Call──→ 10.0.0.1:8080                         │
└─────────────────────────────────────────────────────────────┘
```

### 健康检查机制

| 方式 | 原理 | 优缺点 |
|------|------|--------|
| **心跳 (TTL)** | 服务定时向注册中心发心跳 | 简单，但有延迟 |
| **主动探测** | 注册中心主动调用服务健康接口 | 及时，但增加注册中心负担 |
| **混合模式** | 心跳 + 探测 | 生产常用 |

## 6.2 负载均衡算法详解

### 常见负载均衡算法

```
轮询 (Round Robin):
  请求1 → A, 请求2 → B, 请求3 → C, 请求4 → A, ...
  优点: 简单公平  缺点: 不考虑服务器性能差异

加权轮询 (Weighted Round Robin):
  A(权重3), B(权重1) → A,A,A,B,A,A,A,B,...
  优点: 可根据服务器性能分配  缺点: 权重需要手动配置

随机 (Random):
  随机选择一个服务器
  优点: 简单  缺点: 可能分布不均

最少连接 (Least Connections):
  选择当前连接数最少的服务器
  优点: 动态适应  缺点: 需要维护连接计数

一致性哈希 (Consistent Hash):
  hash(请求key) → 服务器
  优点: 相同key总是打到同一服务器，利于缓存  缺点: 实现复杂
```

### 一致性哈希原理

```
        0
       /|\
      / | \
     /  |  \
   330  |  30    ← 哈希环 (0~360度)
    |   |   |
    C   A   B    ← 服务器映射到环上
    
  请求key哈希后，顺时针找到第一个服务器
  hash("user:123") = 45 → 顺时针找到 B
  
  优势: 增删服务器只影响相邻区间，大部分请求不受影响
```

## 6.3 熔断器 — 防止级联雪崩

```
  服务A → 服务B → 服务C (挂了)

  没有熔断:
    服务A的请求全部堆积在B, B等待C超时
    → B也开始超时 → A也开始超时 → 全部崩溃 (级联雪崩)

  有熔断:
    C连续失败 → B的熔断器打开 → 直接返回错误(不再调用C)
    → A快速得到失败响应 → 不会堆积 → 系统稳定

  熔断器状态机:
    ┌──────┐  连续失败达阈值  ┌──────┐  超时后尝试  ┌──────┐
    │ 关闭  │───────────────→│ 打开  │─────────────→│ 半开  │
    │CLOSED │               │ OPEN  │              │HALF  │
    └──────┘               └──────┘              └──┬───┘
       ↑                                           │
       └────────── 探测成功,恢复 ──────────────────┘
                    探测失败 → 回到OPEN
```
